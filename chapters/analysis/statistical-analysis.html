17  Statistical Analysis

17.1 Statistical Software

  • Excel: Whenever possible, data should be stored in Excel or CSV format, as these are the most accessible formats for sharing across software platforms.

  • R: R has the most flexibility of all our stats software and is very powerful. It also comes with a steep learning curve. However, AI developments have made R dramatically more accessible, so using R is highly encouraged. See the R Workflows section for lab-specific guidance.

  • Mplus: We use Mplus heavily — mostly for analyses including Monte Carlo simulations for assessing statistical power, path analysis, structural equation modeling (SEM), multilevel modeling (MLM), mixture modeling, and approaches that blend these techniques (e.g., multilevel SEM). This is an especially useful package for examining psychometrics. See the Mplus Workflows section for templates and conventions.

  • Stata: A very flexible and comprehensive stats package. See the Stata Workflows section for lab-specific guidance.

  • SPSS: We have mostly used SPSS for data processing and management, data exploration, and basic analyses such as descriptive statistics and OLS regression.

17.2 Using AI for Data Analysis

AI tools like Claude can dramatically accelerate data analysis work, especially for writing and debugging code. Used well, AI is like having a knowledgeable collaborator available at all times. A few principles for using AI effectively and responsibly:

  1. Never share identifiable participant data with AI tools. Paste only anonymized or synthetic examples when asking for help with data-related code.
  2. Always review and understand the code AI produces before running it. Do not run code you cannot explain.
  3. AI output should be treated like a first draft: check it, test it, and own it.
  4. Note AI contributions in your code header as you would note any collaborator’s contribution (e.g., “Analysis code developed with Claude assistance”).

17.2.1 IDEs

An IDE (Integrated Development Environment) is the software application you use to write and run code. The lab’s recommended IDE is Positron, developed by Posit (the makers of RStudio). Positron works well with R, supports Quarto documents, and integrates cleanly with GitHub. It is the successor to RStudio and is what new team members should install. See the Getting Started chapter for installation instructions.

Cursor is an alternative IDE with deeper AI integration that some team members use for more AI-assisted coding workflows.

17.2.2 Claude Code

Claude Code is a command-line AI coding tool from Anthropic that can assist with writing, editing, and explaining code directly in your terminal. It is particularly useful for larger coding tasks where you want AI assistance without leaving your development environment. Claude Code is optional but recommended for team members doing substantial R or Stata work. See the Getting Started chapter for setup guidance.

17.3 Quality Control for Analyses

It is inevitable that at some point we will make mistakes in coding and running analyses, especially when there are many steps involved in producing output. Here is our process for managing quality control:

  1. At the beginning of all code, indicate the code author(s) and contact information.
  2. Add a note in the analysis indicating whether and how you checked that the code worked properly. For example, when using merge code, indicate that you checked N and descriptives and that everything lined up.
  3. Have another team member check your code. Suggested reviewers by software:
    • SPSS: Mike, Joanna
    • Mplus: Mike
    • Stata: Mike
    • R: Mike
  4. When the reviewer checks the code, indicate so in the code header (e.g., “Code reviewed by Michael Strambler”).
  5. After summarizing findings in narrative and/or table format, use AI to check for any misreporting.

17.4 Statistical Language

See the Data Analysis Primer for descriptions of common statistics we use. A few things are worth emphasizing here.

17.4.1 Statistical Significance

Statistical significance is a concept that is often misinterpreted — especially by a lay audience, but also by researchers — as an effect or difference being meaningful. A statistical effect and a meaningful or practical effect can be entirely different things, especially when effects are small. It is better to refer to statistical significance for what it really means: a statistical effect or a statistical difference. For example:

  • The two groups were statistically different.
  • We interpreted a p-value of <.05 as a statistical effect.

17.4.2 Effect Sizes

The real meaning of an effect is expressed through an effect size and its interpretation. An effect size is a statistic that indicates the strength of an effect. This can be a regression coefficient (b or B), mean difference, Cohen’s d, or any other measure that captures an effect’s magnitude. In terms of judging whether an effect size is “big” or “small,” it is best to place it in the context of other research — compare your effect to effects from similar findings (e.g., “The Cohen’s d of .30 is comparable to the findings of Smith (2019) who found effect sizes ranging between .27 and .35”).
