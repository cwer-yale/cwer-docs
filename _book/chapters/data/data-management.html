12  Data Management

Good data management is the foundation of reproducible research. When data are well-organized, clearly documented, and consistently formatted, analyses are easier to run, errors are easier to catch, and handing off work to a teammate is straightforward. This chapter describes CWER conventions for codebooks, variable naming, and missing data. These conventions apply across all projects and all software platforms.

12.1 Codebooks

A codebook is the authoritative record of every variable in a dataset. It documents what each variable measures, how it was collected, what the response options mean, and how missing data are coded. Every CWER project that involves original data collection must have a codebook.

12.1.1 Format

Codebooks are maintained as Excel files (.xlsx) stored in the docs/ folder of the project. Use one sheet per instrument or data source. The file should be named descriptively, for example codebook_care4kids_wave1.xlsx.

12.1.2 Required Columns

Every codebook must include the following columns:

Column Description
variable_name The exact variable name as it appears in the dataset
variable_label A plain-language description of what the variable measures
source Where the variable came from (e.g., survey item, administrative record, derived)
item_text The exact question or item wording presented to respondents, if applicable
response_values The numeric values used to represent response categories
response_labels The text label for each numeric value
missing_codes Any special codes used to represent missing data for this variable
notes Any additional information relevant to using or interpreting this variable

12.1.3 Example Row

variable_name variable_label source item_text response_values response_labels missing_codes notes
sel_selfmgmt_1 Self-management item 1 Teacher survey “This student controls their emotions” 1, 2, 3, 4 Never, Sometimes, Often, Always -99, -98 Reverse coded for composite

12.1.4 Keeping Codebooks Current

A codebook is only useful if it is kept up to date. Update it whenever:

  • A new variable is added to the dataset
  • A variable is recoded or transformed
  • A response category changes meaning across waves
  • A missing data code changes

Treat the codebook as part of the analysis, not an afterthought. Ideally, draft the codebook before data collection begins — this forces clarity about what you are measuring and how you will analyze it.

12.2 File and Folder Naming Conventions

Consistent naming makes projects easier to navigate, reduces errors, and ensures files work correctly across different operating systems and software. CWER follows two conventions depending on context:

  • Dashes for file and folder names — dashes are URL-friendly and the standard for web-facing files. When Quarto renders documents, file names become URLs, and dashes produce cleaner, more readable links than underscores.
  • Underscores for variable names — snake_case is the standard for R, Stata, and data science broadly. See the Variable Naming Conventions section above.
Context Convention Example
File names kebab-case (dashes) data-wrangling.qmd
Folder names kebab-case (dashes) cwer-docs/
Variable names snake_case (underscores) sel_total_mean
R object names snake_case (underscores) data_cleaned
Data files snake_case (underscores) wave1_cleaned.rds

Note that data files use underscores rather than dashes. This is because data files are read programmatically by R, Stata, and SPSS, where underscores are safer and more conventional than dashes.

12.2.1 Rules for All File and Folder Names

  • Use lowercase only. wave1-cleaned.rds, not Wave1-Cleaned.rds
  • Be descriptive. wave1-provider-survey.csv tells you what the file is; data.csv does not
  • Avoid spaces. Spaces in file names cause problems in terminal commands and some software. Use dashes or underscores instead
  • Avoid special characters. No parentheses, slashes, ampersands, or symbols of any kind
  • No date prefixes on individual files. Do not use names like analysis-2025-10-01.qmd — let Git track versions instead. If you genuinely need multiple dated versions to coexist (e.g., comparing model runs), use a dated subfolder inside output/ rather than dating the file itself

12.2.2 Rules for Script Files

Number scripts to reflect pipeline order:

code/
├── 01-data-cleaning.qmd
├── 02-descriptives.qmd
├── 03-mlm-analysis.qmd
└── 04-tables-figures.qmd

The two-digit prefix ensures scripts appear in order when listed. When you add a new script, use the next available number. If you retire a script, do not renumber the remaining ones — leave gaps rather than creating confusion about which outputs correspond to which script.

12.2.3 Rules for Output Files

Name output files to reflect their content and the script that produced them:

  • output/figures/sel-by-grade-boxplot.png — descriptive, clear
  • output/tables/wave1-mlm-results.docx — clear what analysis and wave
  • Not: output/figures/figure-final2.png — tells you nothing

When a script produces multiple outputs of the same type, use a consistent naming pattern:

output/figures/
├── fig1-sel-distribution.png
├── fig2-sel-by-grade.png
└── fig3-sel-by-school.png

12.3 Variable Naming Conventions

Consistent variable names make datasets easier to work with across software platforms, reduce errors, and make analysis code more readable. The following conventions apply to all CWER datasets.

12.3.1 Why Snake_case?

CWER uses snake_case — lowercase letters with underscores between words — as the standard naming convention. For example: sel_selfmgmt_1, teacher_id, rdg_score_z.

This is a deliberate choice over alternatives like camelCase (SelSelfmgmt1) or PascalCase (SelSelfMgmt1) for several reasons:

  • Cross-software portability. Snake_case works cleanly across R, Stata, SPSS, and CSV files. Stata is case-sensitive, which means RaceEth and raceeth are treated as different variables — a common source of hard-to-find errors. Lowercase names eliminate this risk entirely.
  • Readability. sel_selfmgmt_mean is easier to parse at a glance than SelSelfmgmtMean, especially for longer names with multiple components.
  • Industry standard. Snake_case is the dominant convention in data science — it is what R’s tidyverse, Python, and most database systems use. RAs with any quantitative background will already be familiar with it.

12.3.2 General Rules

  • Use lowercase letters only. sel_score, not SEL_Score or Sel_Score.
  • Use underscores to separate words. teacher_rating, not teacherrating or teacher.rating.
  • Keep names short but meaningful. Aim for 8 characters or fewer when possible for compatibility with older software, but never sacrifice clarity for brevity. A name like rdg_score is better than rs and better than reading_comprehension_total_score.
  • Avoid spaces and special characters. No spaces, hyphens, slashes, parentheses, or symbols of any kind.
  • Never start a variable name with a number. wave1_n is valid; 1_wave_n is not.

12.3.3 Naming Structure

For survey and scale items, use the following structure:

[construct]_[subscale]_[item number]

Examples: - sel_selfmgmt_1 — SEL scale, self-management subscale, item 1 - sel_selfmgmt_2 — SEL scale, self-management subscale, item 2 - sel_socaware_1 — SEL scale, social awareness subscale, item 1

For composite scores and derived variables:

[construct]_[subscale]_[statistic or type]

Examples: - sel_selfmgmt_mean — mean of self-management items - sel_total_mean — mean across all SEL items - rdg_pre_score — reading pre-test score - rdg_post_score — reading post-test score

For demographic and administrative variables, use clear, plain names:

  • age_years — age in years
  • grade — grade level
  • gender — gender
  • race_eth — race/ethnicity
  • school_id — school identifier
  • teacher_id — teacher identifier
  • student_id — student identifier

12.3.4 ID Variables

Every dataset must include at least one ID variable that uniquely identifies each row. Follow these rules for ID variables:

  • Name ID variables with the unit of observation followed by _id: student_id, teacher_id, school_id, provider_id
  • ID values should be numeric where possible
  • Never use names, initials, birthdates, or other personally identifiable information as ID values
  • Document the ID scheme in the codebook, including how IDs were assigned and whether they are consistent across waves or datasets

12.3.5 Wave and Time Variables

For longitudinal projects, include a wave or time indicator:

  • wave — numeric wave indicator (1, 2, 3)
  • time — numeric time point
  • year — academic or calendar year
  • season — fall, spring (use text labels)

When variable names need to distinguish across waves, append the wave to the end:

  • sel_total_mean_w1 — SEL total mean, Wave 1
  • sel_total_mean_w2 — SEL total mean, Wave 2

12.3.6 Recoded and Derived Variables

When you recode or derive a variable from an existing one, create a new variable rather than overwriting the original. Append a suffix to indicate the transformation:

Suffix Meaning Example
_r Reverse coded sel_item3_r
_c Centered age_years_c
_z Standardized (z-score) rdg_score_z
_log Log transformed income_log
_cat Categorical version of a continuous variable age_cat
_imp Imputed rdg_score_imp

Never overwrite raw variables. Always create a new variable and document it in the codebook.

12.3.7 Variable Names in Mplus

Mplus imposes an 8-character limit on variable names, which creates a tension with snake_case. An underscore uses one of those 8 characters, leaving only 7 for meaningful content. For example, sel_sm_1 hits the limit exactly, while longer names like sel_selfmgmt_1 (14 characters) are simply not valid in Mplus syntax.

The recommended approach is to maintain full snake_case names in your master datasets and remap to short Mplus-compatible names only when preparing Mplus input files. The MplusAutomation package in R makes this straightforward — you can keep descriptive names in R, write a short name mapping when exporting to Mplus, and map results back to full names when importing output. This way your master data and R code stay readable, and your Mplus syntax stays within the character limit.

When constructing short Mplus names, follow these guidelines:

  • Drop underscores and abbreviate: sel_selfmgmt_1 becomes selsm1
  • Keep names as interpretable as possible within the limit
  • Document the mapping between full names and Mplus names in your codebook or script header

12.4 Missing Data Codes

Missing data arise for different reasons, and distinguishing between them is important for accurate analysis and documentation. CWER uses a two-step approach: numeric codes for storage, and software-native formats for analysis.

12.4.1 Step 1: Storage Codes

Use the following numeric codes in all raw and cleaned data files across all projects. These codes travel across software platforms — a CSV file with -99 is readable by R, Stata, SPSS, and Mplus alike.

Code Meaning
-99 Item not answered / respondent skipped
-98 Not applicable (item did not apply to this respondent)
-97 Data not available (data exist but were not collected or could not be obtained)
-96 Removed during data cleaning (e.g., identified as invalid response)

These codes should be applied consistently within a project and documented in the codebook for each variable that uses them.

Never use 0 to represent missing data unless 0 is a meaningful response value that is completely distinct from missingness — and even then, use caution and document it explicitly.

12.4.2 Step 2: Analysis-Ready Formats

Before running any analysis, recode CWER storage codes to each software’s native missing data format. This is a required step — failing to recode is one of the most common sources of errors in quantitative research. A -99 included in a mean calculation will silently produce a wrong answer.

R — recode to NA:

data <- data |>
  mutate(across(everything(), ~ if_else(. %in% c(-99, -98, -97, -96), NA, .)))

Stata — recode to .:

foreach var of varlist _all {
  recode `var' (-99 -98 -97 -96 = .)
}

Mplus — declare in the MISSING statement:

VARIABLE:
  MISSING = ALL(-99 -98 -97 -96);

SPSS — declare as user-missing values:

MISSING VALUES ALL (-99, -98, -97, -96).

Do this recoding step in your cleaning script (code/01-data-cleaning.qmd or equivalent) so that all downstream analysis scripts work with properly coded missing data from the start.

12.4.3 Documenting Missingness

For every dataset, produce a missing data summary as part of your cleaning script. At minimum, report the number and percentage of missing values for each variable. Flag any variables with more than 10% missing for discussion with Mike or Joanna before proceeding with analysis.

In R:

data |>
  summarise(across(everything(), ~ sum(is.na(.)))) |>
  pivot_longer(everything(), names_to = "variable", values_to = "n_missing") |>
  mutate(pct_missing = round(n_missing / nrow(data) * 100, 1)) |>
  filter(n_missing > 0) |>
  arrange(desc(n_missing))

12.5 Data Cleaning Standards

Data cleaning scripts (code/01_data_cleaning.qmd or equivalent) should follow these standards:

  1. Never modify raw data. Read from data/raw/, write cleaned files to data/cleaned/.
  2. Document every decision. When you recode, drop, or transform data, add a comment in the code explaining why.
  3. Check merges carefully. After any merge, verify that the row count and key variables look as expected. Document the check in the script header.
  4. Produce a cleaning log. At the end of the cleaning script, output a summary of what was done: original N, final N, variables added, variables recoded, missing data summary.
  5. Have your cleaning code reviewed. Follow the quality control process described in the Statistical Analysis chapter.

12.6 A Note on Sensitive Data

Variables that could identify participants — names, addresses, birthdates, school names in small samples, teacher names — require special handling. See the Being a Team Member chapter for data security requirements. As a general rule:

  • Strip identifying variables from analysis datasets as early as possible in the cleaning process
  • Keep a separate, secure crosswalk file linking IDs to identifying information if re-identification is ever needed
  • Store crosswalk files in Yale OneDrive or Storage @ Yale, never in the project GitHub repository
